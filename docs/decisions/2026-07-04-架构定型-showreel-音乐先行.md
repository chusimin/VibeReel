# 架构定型 —— 从"三类型平台"收窄到"showreel 品味引擎"

> 本文档记录 2026-07-04 一次对话里，VibeReel 从"多类型视频生成平台"转向"showreel 品味引擎"的完整决策过程。
> 重点不是新范围，是**为什么这么改**、**每一步骤谁负责决定什么**、**哪些抽象保留、哪些收窄**。
> PRD 主文档将据此重写为 v0.4。

---

## 0. TL;DR

- **v1 只做 showreel**（教学 / 科普推到 v2）。四件套抽象保留，视频类型枚举从 3 收到 1。
- **流程从 4 闸门扩到 6 闸门**：方向 → **音乐** → **脚本** → 草稿 → 单镜 → 终检。
- **"音乐先行"**：音乐一旦定，BPM / 时长 / 拍点即成硬约束；后续每一镜时长必须落在拍点上。
- **v1 音乐 = 免费库匹配为主，AI 生成排 v1.5**（AI 生成的音乐目前节奏跟不上，像歌曲配乐不像 showreel BGM）。
- **26 个拆解维度**（14 元素 + 6 结构 + 6 技术）作为参考视频拆解的标准。
- **三层品味库（skill）**：原子 skill / 模板 skill / 反例 skill。
- **一次生成必须能上线可用**（不是"过得去"，是"直接发推")—— 这是从此以后一切设计的第一硬约束。

---

## 1. 决策一览

### 1.1 定位收窄（保留抽象）

**决策**：v1 视频类型从 `showreel | teaching | popsci` 收到只做 `showreel`。

**保留抽象**：
- 公共骨架（方向 → 脚本 → 草稿 → 单镜 → 合成）仍然存在
- 四件套（知识包 / 风格包 / 闸门配置 / QA 规则）**代码结构保留**，v1 只装 showreel 一套
- `videoType` 字段保留，未来加类型 = 加一套四件套，不改主流程

**为什么**：
- 品质焦点：一次做透一类，不做"三类各七成"
- 独立开发者的宣传视频 90% 是 showreel 类型（品牌 motion / 产品秀）
- teaching / popsci 需要讲稿、TTS、字幕、更复杂的信息设计，v1 想清楚 showreel 再扩

**代价 / 认知**：
- PRD v0.3 里花力气做的多类型抽象**只是纸面收益**，v1 用不上；但代码结构保留意味着 v2 加类型成本低
- 明确告诉朋友们："这个工具目前只擅长做产品宣传短片，教学/科普不好"

### 1.2 一次生成即上线可用（第一硬约束）

**决策**：v1 的验收标准从"可用"升级为"**一次生成必须直接可发推**"。

**含义**：
- 不允许"生成完再手动改视频"—— 用户在闸门内改完就是最终版
- 不允许"90% 好但有 1 处丑"—— 一处丑就是全片丑
- **闸门是保证质量的最后关卡**，AI 输出必须在每个闸门通过硬校验才能进下一步

**为什么**：
- 用户是独立开发者，不是设计师，改视频这一步他们无法完成
- "还能凑合"的视频对独立开发者的品牌**是负资产**（宁可不发也不能发丑的）
- 只有这个约束在，Agent 才有压力持续调优

---

### 1.3 六闸门流程（音乐先行）

**决策**：流程从 4 闸门扩为 6 闸门。

```
[输入]         [闸门①]        [闸门②]        [闸门③]        [闸门④]        [闸门⑤]        [闸门⑥]
想法/URL/     方向探索        音乐选择        脚本编辑        草稿确认        单镜确认        终检
代码包    →   (2-3 卡)    →  (5 首候选)  →  (增/删/改镜) →  (moodboard    →  (逐镜)     →  (成片)
                                                             + 低保真)
```

**每个闸门的产物**：

| 闸门 | AI 产出 | 用户操作 | 产物落入 project.json |
|---|---|---|---|
| ① 方向 | 2-3 张方向卡（含 look/palette/pacing/music_hint） | 三选一或打回 | `concepts[]` + `chosenConcept` |
| ② 音乐 | 5 首候选（AI 根据方向卡生成 query，并行查库） | 选一首或换一批 | `music: {url, bpm, beats[], duration, source}` |
| ③ 脚本 | N 镜文字大纲（时长必须踩拍） | 增镜/删镜/改镜文字/拖拽排序 | `scenes[].{role, onScreenText, durationSec, beatAlign}` |
| ④ 草稿 | 每镜情绪 moodboard + 低保真真渲缩略 | 整体确认或逐镜打回 | `scenes[].{draftMood, draftLoFi}` |
| ⑤ 单镜 | 每镜真渲 mp4 | 逐镜 👍/👎 | `scenes[].mp4` |
| ⑥ 终检 | 合成成片 + QA 报告 | 下载或返回改 | `outputs.mp4` |

**关键性质**：
- **闸门②之后，视频总时长即锁定**（= 音乐时长）
- **闸门③每镜 durationSec 必须是"两拍间距的整数倍"**（beatAlign 字段记录该镜起止是第几拍到第几拍）
- **闸门④的"低保真真渲"用的模板 = 闸门⑤真渲用的同一模板**，只是分辨率降级；确保用户在草稿看到什么，正片就是什么
- **每个闸门都有"返回上一闸门"能力**，但返回会作废下游产物

**为什么加"音乐先行"**：
- 真实设计师就是先定 BGM 再切片，不这么做的成片一定"业余感"
- 拍点是硬约束 = AI 判断收敛（不用凭空想时长）
- 音乐库匹配只算一次，脚本重来不影响音乐，省时省钱

**为什么脚本步独立成闸门**：
- 用户在脚本层的修改（增删改镜）后面改就来不及
- 骨架错了，画面再好也白搭；先让用户对骨架签字
- 分离"文字判断"和"画面判断"减轻用户认知负担

### 1.4 音乐库策略

**决策**：
- **v1**：免费库匹配（Pixabay + Uppbeat 二选一 / 或全接）+ **本地精选池**兜底
- **v1.5**：加 CLAP embedding 排序（提升匹配准确度）
- **v2**：接 AI 音乐生成（v1 试过 Suno / Udio，节奏跟不上，像歌曲配乐）

**三层匹配架构**：
1. AI 从"方向卡"翻译成结构化 query（mood / bpm_range / genre / energy_curve / duration_target）
2. 并行查库 → 拿 20-30 候选 → embedding 排序（v1.5）
3. 拿到 Top-5 后**从音频文件重新精确计算 BPM 和拍点**（不信任平台标注）

**本地精选池**：
- 目标 200 首（v1 起步 30 首足够）
- 都是我们审过的、能出好片的曲子
- 用户先匹配精选池，命中率高就不查外部 API（省 API 调用 + 保底质量）
- 精选池按"showreel 音乐性格"分类（如：确定推进 / 神秘氛围 / 兴奋爆发 / 克制高级 / 温暖回忆）

**为什么先建精选池**：
- 外部 API 标签质量参差不齐，精选池是我们的"品味的最后防线"
- 未来接 AI 生成的音乐也可以塞进精选池（人工过滤后）

---

### 1.5 26 个拆解维度（参考视频拆解标准）

**决策**：所有参考视频按 3 层共 26 个维度拆解，产出统一格式的 `.md` + `beat_map.json` + `shot_timeline.json`。

**层 1 · 元素维度（14 项）**

| # | 维度 | 拆什么 |
|---|---|---|
| 1 | 音乐 | BPM / 情绪 / 结构(intro-build-drop-outro) / drop 时间点 |
| 2 | 节奏 | 每镜时长分布 / 是否卡拍 / 长短镜比例 / 快慢切换点 |
| 3 | 氛围 | 明暗曲线 / 冷暖倾向 / 密度(简约/繁复) |
| 4 | 颜色 | 主色板 3-5 hex / 单色 or 撞色 / 色温 / 对比度 |
| 5 | 品牌元素 | logo 次数/时长/位置/入场方式 |
| 6 | 产品画面 | UI 截图数 / 展示方式(浮动/mockup/融入场景) |
| 7 | 产品代码 | 代码画面镜数 / 呈现方式(打字机/diff/IDE 截图) |
| 8 | 字体 | 主字体 / 字重 / kinetic-text / 中英文混排 |
| 9 | 比例 | 16:9 / 9:16 / 1:1 |
| 10 | 动态效果 | 每镜核心动效动词表 |
| 11 | 转场 | 每两镜之间的转场类型 |
| 12 | 构图 | 每镜构图(居中/三分/满屏/负空间比例) |
| 13 | 文字节奏 | 文字什么时候出现、停留多久、多少字/镜 |
| 14 | 视觉密度 | 整片密度曲线(极简/信息密) |

**层 2 · 结构维度（6 项）**

| # | 维度 | 拆什么 |
|---|---|---|
| S1 | 起承转合 | 开场/铺陈/高潮/收束的时间分布 |
| S2 | 叙事弧线 | 卖点罗列 / 故事线 / 情绪堆叠 |
| S3 | 重复与变奏 | 反复出现的视觉母题 |
| S4 | 信息密度曲线 | 什么时候放松/紧张 |
| S5 | 首帧与末帧 | 第一秒 & 最后一秒长什么样(海报感) |
| S6 | hook 时刻 | 前 3 秒抓人的是什么 |

**层 3 · 技术维度（6 项）**

| # | 维度 | 拆什么 |
|---|---|---|
| T1 | 分辨率与帧率 | 1080p/30 vs 4K/60 感官差异 |
| T2 | motion blur | 有没有 / 强度 |
| T3 | grain / texture | 胶片颗粒 / 纹理叠加 |
| T4 | 音效层 | BGM 之外有没有 whoosh/click/impact |
| T5 | 调色 | LUT 倾向 / 高光溢出 / 阴影提升 |
| T6 | easing 曲线 | easeOut / easeInOut / cubic-bezier |

**拆解工具**：
- beat 检测：`aubio`
- shot 切分：`ffmpeg -vf select='gt(scene,0.3)'`
- 帧采样：`ffmpeg -vf fps=1/0.3`
- 主色提取：`ffmpeg` + `ImageMagick` histogram
- 逐镜视觉打标：**当前 Agent 层看不到图像**，需要人工陪打 / 视觉 API / 抽帧后传图

**当前限制**：v1 拆解要么用户陪打（一条 17s 视频 ~5 镜、~20 分钟），要么调视觉 API。**待定**。

### 1.6 三层品味库（skill）

**决策**：所有品味沉淀为三层 skill，作为 Agent 输出的知识底座。

| 层 | 数量 | 内容 | 更新方式 |
|---|---|---|---|
| **原子 skill** | 26 个（对应 26 拆解维度） | 每个维度"什么是好的、什么是差的"，带具体参考视频时间戳例子 | 每拆一条新参考视频，反向补充所有相关原子 skill |
| **模板 skill** | 起步 3-5 个（每条参考视频 1 个） | 一条视频的完整配方(BPM 128、5 镜、颜色 #xxx、每镜动效...)。用户可选"用这个模板骨架" | 加参考视频 = 加模板 |
| **反例 skill** | 1 个 | 什么样的 showreel 是丑的(节奏拖沓、拼贴感、字太多、色不统一...) | 每次生成失败/被打回，反例库补一条 |

**加载策略**：
- v1：手动 include 到 prompt（业务代码里直接拼字符串）
- v1.5：切 pi Agent + skill 系统自动加载

**为什么现在就落 skill 结构**：
- 换 pi Agent 不换 skill，skill 是持久资产
- 你说"以后我会看到很多漂亮示例就想补进来"—— 必须有一个稳定的沉淀格式，否则一堆散笔记
- 反例 skill 特别重要：Agent 光知道好不够，要知道边界

---

### 1.7 上传素材"用途标签"

**决策**：每个用户上传的素材必须选一个用途标签。

| 标签 | 含义 | 影响 |
|---|---|---|
| **必现（must-appear）** | 这张一定要在正片里出现至少一次 | 分镜阶段 Agent 必须为它派一镜 |
| **可参考（may-use）** | AI 觉得合适就用，不用也行 | 分镜阶段 Agent 视需要引用 |
| **只调性（tone-only）** | 不出现在正片，只作为色调/氛围参考 | 只喂给方向/音乐/草稿阶段 |

**为什么**：
- 现在的问题是"上传了但没进正片"，用户失望
- 强制标签让用户预期与产出对齐
- Agent 有明确指令知道该拿这张图干什么

---

### 1.8 底层技术架构决策（暂缓 pi Agent 切换）

**决策**：
- **v1 保持现有 `complete()` 单轮 completion 架构**
- 优先修数据串联 bug（讲稿 → 脚本落盘、refs 校验、上传图打标签串到渲染）
- Skill 库沉淀到有 3-5 个模板 skill + 10+ 个原子 skill 后，**再切 pi Agent**（这时切换有质变）

**为什么**：
- 换 Agent 但没有 skill 内容 = 换了个仓库没有书
- 数据链 bug 不是 Agent 的问题，是 orchestrator 的问题，先修
- pi Agent 的最大价值是 skill 自动加载 + 工具化，前者要 skill 内容准备好

**并发**：
- v1 目标 10 人使用，单机足够
- 项目结构留好并发路径（数据/项目分文件夹、锁按项目 id），未来加队列服务即可

## 4. 执行顺序（本决策文档拍板后）

1. ⬜ **先拆 1 条参考视频**（Aftermagics datalyr 17s 那条），产出 `docs/references/*.md` 作为样板
2. ⬜ 你审样板，确认拆解格式满意
3. ⬜ 剩余 6 条参考视频批量拆
4. ⬜ 拆解产出反向沉淀为 26 个原子 skill + 7 个模板 skill + 1 个反例 skill 底稿
5. ⬜ 重写 PRD 到 v0.4
6. ⬜ 修数据串联 bug（讲稿→脚本落盘、refs 校验、上传图 usage 标签）
7. ⬜ 扩流程为 6 闸门（新增音乐闸门 + 脚本闸门）
8. ⬜ 接免费音乐库（Pixabay / Uppbeat）+ 精选池起步 30 首
9. ⬜ 上传图真进正片（still-kenburns 后端接 usage=必现 的素材）
10. ⬜ 草稿改为"moodboard + 低保真真渲"
11. ⬜ v1 打磨到"一次生成可发推"，观察 2 周
12. ⬜ 切 pi Agent + skill 自动加载
