# 从 skill 到 VibeReel：一次架构决策的完整过程

> 记录 2026-06-12 一次对话里，从「想把 GSAP 动效库补进 skill」出发，如何一步步推到「VibeReel 做成多类型抽象 + 多后端的视频生成平台」，并把 PRD 从 showreel 薄壳重写成 v1 完整平台。
> 重点不是结论，是**每个转折点为什么这么决定**，以及走过的弯路。

---

## 一句话概括

一个「想补个动效库」的小需求，被一路追问，最后变成了一次平台级的架构定型。关键不是加了什么库，是想清楚了**三件事各在哪一层**：动效手艺（知识）、多后端渲染（引擎）、产品编排（VibeReel）。

---

## 决策轨迹

### 0. 起点
背景：在持续打磨 `vibe-motion-video` skill（Remotion 视频生成引擎）。当天先收了两件小事：
- 把方法论长文《AI 做视频是写不是画》重写、配 3 张黑白极简 excalidraw 图，输出到 Obsidian `30_产出/`。
- 把引擎**默认画幅从 9:16 改成横版 16:9**（竖屏仍可显式 `--platform douyin`）。

### 1. GSAP skills 对比
- **起**：发现 GreenSock 官方 `gsap-skills`，直觉是「它的动效库 + 知识库非常适合补给我，这样 showreel 能直接用」。
- **查**：下载分析。GSAP skills = 教 AI 写对 GSAP（**网页 JS 动画库**）的官方知识包，8 个子 skill，`npx skills add` 跨 40+ agent。
- **得**：两者同类（都是「教 AI 做动效」的 skill）但不在一层 —— **GSAP 管「动效代码怎么写对」，vibe-motion 管「一条视频怎么做好」**。互补不竞争；ease / stagger / 响应式等手艺两边相通。

### 2. 关键纠正：GSAP 库不能直接用 ⚠️
- **问**：能不能把 GSAP 库直接塞进 skill 用？还能用别的库吗（我只关心成片）？
- **答（诚实泼冷水）**：GSAP 是**运行时按真实时钟实时播放**的库；Remotion 是**跳到第 N 帧截图**的逐帧渲染。时间模型冲突，硬塞会让动画不动 / 错位，**反而毁成片**。
- **决策**：
  - GSAP 当**知识 / 手艺**吸收（ease 词典、SplitText、Flip、MorphSVG 等手法 → 提炼进动效宪法），**不当依赖库搬**。
  - 要扩表现力，用 **Remotion 同源生态**：`@remotion/transitions`、`@remotion/lottie`、`paths/shapes`、`motion-blur`。
  - 一句话：**GSAP 的脑子可以学，手不能搬**。

### 3. 思敏的洞察：不同分镜不同实现 💡
- **问**：有没有另一种思路，不同分镜用不同方式实现？
- **想透**：这正好兑现 skill 的「图纸 / 引擎解耦」理念，把「整片换引擎」升级成「**逐镜选引擎**」。而且**把 GSAP 救回来了** —— 它不塞进 Remotion，而是当一个**平级后端**（GSAP 镜头独立 headless 渲成 mp4 段）。
  - 底层道理：**成片是 mp4，任何能吐出 mp4 段的工具都能当后端，最后 ffmpeg 拼**。前面那个时间模型冲突，在「各渲各的、最后拼」里直接消失。
- **决策**：多后端架构。storyboard 每个 scene 声明 `renderer`；render 分发；assemble 统一合成。
- **冷水**：跨后端**视觉统一**（色板 / 字体 / 安全区）是最大挑战，否则「几条片硬拼」的拼贴感。这成了后来 PRD 里的**硬护栏**。

### 4. 从 skill 到平台
- **愿景**：要做一个平台 —— 模板抽象 + Agent 判断用哪些流程 / 库 / 工具 + 每分镜单独实现 + 合成 + 配音。问：要不要复制 skill 到新项目？是不是不止一个 skill？
- **厘清**：
  - skill ≠ 平台；「复制粘贴源码」是反模式（两份漂移），正解是「**复用引擎**」。
  - **发现两天前已写好 VibeReel PRD**，且早就正确定了「新项目 + 复用引擎不复制」。今天的纠结，两天前的自己已答过。
  - 关键提醒：**多后端是引擎层升级，VibeReel 是产品壳，两层解耦**，别让大架构冲垮 PRD 里定的薄验证节奏。
- **决策（选定形态）**：**新建 VibeReel 产品**（`claude-agent-sdk` 编排 + 子进程调 `vibemotion` 引擎），引擎复用不复制；多 skill 拆分用于**引擎层**组织。

### 5. PRD 体检 + 一串拍板
- **体检**：PRD 写得扎实；核对 styleId / 平台**无对齐 bug**。挑出：平台默认变了要钉死 `douyin`、edge-tts 要兜底、静帧从哪来、等待预期、生成空窗等。
- **逐条拍板**：
  - **画幅**：可选 16:9 / 9:16 / 1:1，开始前**引导用户尽量上传素材**。
  - **静帧**：改用 **gpt-image 出每镜草稿 → 确认方向 → 再渲正片**（两阶段）。
  - **输出**：**逐分镜流式**，渲一个展示一个；**生成态要好好设计**；暂不显示时长。
  - **数据**：项目**元数据驱动** + **分镜级版本记录**。
  - **多后端**：现在就做（认为提质一大截）。
  - **范围**：三类型（showreel / 教学 / 科普）一起，**多类型抽象**，PRD 按完整版设计、**逐场景实现**。
  - **Key**：Anthropic BYO 不变，OpenAI / 生成式平台配置；**以后付费**。

### 6. PRD 重写为 v0.2
- 整篇重写：**公共骨架**（方向 → 分镜 → 分段 → 终检）+ **四件套**（知识包 / 风格包 / 闸门配置 / QA 规则）。
- 多后端 v1 接 4 个：`remotion` / `generative` / `lottie` / `still-kenburns`。
- 草稿→正片两阶段、多画幅、元数据 + 版本、素材上传、生成态、edge-tts 兜底、`COOKIE_SECRET` 入 env。
- **挖出两个缺口**：教学四件套不全（需引擎补 playbook + audio）、1:1 画幅引擎无对应平台。

## 关键决策汇总

| 决策 | 选择 | 理由 |
|---|---|---|
| GSAP 怎么用 | 学知识，不搬库 | 运行时库与 Remotion 逐帧渲染时间模型冲突 |
| 扩表现力 | Remotion 同源生态 + 多后端 | 兼容、不毁成片 |
| 不同分镜 | 各选后端，各渲 mp4，最后拼 | 兑现「图纸/引擎解耦」，GSAP 可作平级后端 |
| 平台形态 | 新建 VibeReel，复用引擎不复制 | 避免两份源码漂移 |
| 层次关系 | 引擎升级（多后端）/ 产品壳（VibeReel）解耦 | 引擎变强产品自动受益，互不拖累 |
| 范围 | 三类型 + 多类型抽象，逐场景实现 | 一套骨架装三类，差异收进四件套 |
| 静帧 | gpt-image 草稿 → 正片两阶段 | 分镜阶段给方向参考，不渲昂贵正片 |
| Key | Anthropic BYO + 平台配 OpenAI/生成式 | 最小改动，付费排 v2 |

## 一条贯穿始终的方法
每一步都在问同一个问题：**这件事到底属于哪一层？** 动效手艺属知识层、多后端属引擎层、编排属产品层。把层分对了，「复制还是复用」「一个还是多个 skill」「库能不能直接用」这些纠结，答案就自己浮出来了。
